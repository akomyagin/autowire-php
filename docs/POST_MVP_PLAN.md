# AutowirePHP — POST_MVP_PLAN

Всё ниже **вынесено за скоуп MVP** (см. PLAN.md → «Что осознанно режем в v1»).
Это nice-to-have направления после того, как пять MVP-этапов закрыты и покрыты
тестами. Порядок — не обязательство, а список кандидатов.

## 1. PHP 8 attributes auto-configuration

Конфигурация резолвинга через атрибуты вместо (или в дополнение к) явному
`bind()`:

- `#[Inject]` / `#[Named('...')]` на параметрах конструктора для явного выбора
  реализации при неоднозначном интерфейсе.
- `#[Singleton]` на классе как декларативная альтернатива lifecycle-регистрации
  из Этапа 4.
- Чтение через `ReflectionParameter::getAttributes()` /
  `ReflectionClass::getAttributes()`.

Почему после MVP: усложняет резолвер и вводит второй источник правды поверх
runtime-reflection; сначала должен устояться базовый механизм.

## 2. Компиляция / кеширование графа зависимостей для прода

Runtime-reflection на каждый `get()` — удобно для разработки, дорого под
нагрузкой:

- Один раз проанализировать граф и сгенерировать «скомпилированный» контейнер
  (PHP-файл с прямыми `new`-вызовами) либо сериализованный план резолвинга.
- Инвалидация кеша при изменении биндингов/классов.
- Замер: сравнить cold reflection vs compiled на осмысленном графе.

Почему после MVP: это оптимизация ради прода, а проект — learning-билдер;
преждевременная компиляция скрыла бы саму механику Reflection, ради которой
проект и затевался.

## 3. PSR-11 совместимость

Реализовать `Psr\Container\ContainerInterface`:

- `get(string $id): mixed` и `has(string $id): bool` по контракту PSR-11.
- Маппинг внутренней иерархии исключений на
  `Psr\Container\NotFoundExceptionInterface` /
  `ContainerExceptionInterface`.
- Добавить `psr/container` в `require`.

Почему после MVP: даёт интероперабельность с экосистемой (фреймворки, middleware),
но требует стабилизировать публичный контракт `get`/`has`, который до Этапа 5 ещё
меняется.

## Прочие кандидаты (backlog, без обязательств)

- Contextual binding (разная реализация интерфейса в зависимости от потребителя).
- Factory-биндинги (`bind` на замыкание-фабрику вместо class-string).
- Теговые коллекции для variadic-параметров (осмысленная альтернатива «пустому
  набору» из Этапа 5).
- Отложенный/ленивый прокси-резолвинг тяжёлых зависимостей.
