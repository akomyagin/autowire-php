# AutowirePHP — PLAN

## Видение

**AutowirePHP** — framework-agnostic DI-контейнер для PHP с автовайрингом на
базе Reflection API. Библиотека резолвит граф зависимостей автоматически, не
требуя ручной регистрации каждого класса: контейнер сам читает конструктор,
разбирает type-hint'ы параметров и рекурсивно создаёт зависимости.

Проект — не обёртка над готовым решением и не production-фреймворк, а
learning-билдер PHP-fluency. Это **первый проект на чистом PHP** в портфеле
(до этого PHP только *анализировался* Kotlin-плагином ORM N+1 Radar, но не
писался), поэтому упор — на глубокое понимание Reflection API и на одну
по-настоящему нетривиальную техническую задачу: детект циклических
зависимостей, которые проходят **через интерфейсы** при ленивом резолвинге.

Не бизнес-идея: экономика не оценивается, бюджет $0, цель — искренняя
техническая сложность и обучение.

## Ключевая техническая сложность

Наивный детект цикла хранит «список посещённых конкретных классов». Но при
биндинге `interface -> implementation`, который резолвится **лениво**, цикл
может идти через интерфейсную косвенность: `A` зависит от интерфейса `IB`,
`IB` биндится в `B`, `B` зависит от `IA`, `IA` биндится в `A`. Трекинг
конкретных классов такой цикл не ловит, пока не «размотает» биндинги. Правильный
подход — resolution-stack: отслеживать текущий **путь резолвинга** (и абстрактные
id, и конкретные классы), а не глобальный visited-set. Это главная задача проекта
(Этап 3) и держится отдельным этапом с плотным тестовым покрытием.

## Архитектура (верхнеуровнево)

- **`Container`** — единственная публичная точка входа. Хранит биндинги
  `abstract -> concrete`, кеш singleton-инстансов, реестр lifecycle-меток.
- **Resolver** (логика внутри контейнера) — через `ReflectionClass` /
  `ReflectionParameter` читает конструктор, рекурсивно резолвит зависимости,
  ведёт resolution-stack для детекта циклов.
- **Исключения** — типизированная иерархия: не найден биндинг, класс
  неинстанцируемый, обнаружен цикл (с человекочитаемой цепочкой), неразрешимый
  параметр конструктора.

Никаких внешних сервисов, БД, Docker — чистая composer-библиотека, только
runtime Reflection. Компиляция/кеширование графа для прода в MVP **не входит**.

## Этапы (MVP)

| Этап | Содержание |
|---|---|
| **Этап 1** | Базовый `Container` + ручная регистрация биндингов `interface -> implementation` без autowiring |
| **Этап 2** | Reflection-based автовайринг конструктора: рекурсивный резолвинг type-hinted параметров |
| **Этап 3** | **Детект циклических зависимостей через интерфейсы при ленивом резолвинге** (главная задача, отдельный этап, плотное покрытие тестами) |
| **Этап 4** | Lifecycle-контроль: singleton vs transient |
| **Этап 5** | Edge cases конструктора: union types, nullable, default-значения, variadic |

Детальная разбивка каждого этапа — в [`TECHNICAL_PLAN.md`](TECHNICAL_PLAN.md).
