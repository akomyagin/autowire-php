# AutowirePHP — TECHNICAL_PLAN

Детальный технический план: стек, архитектура, разбивка по Этапам с
критериями готовности (Definition of Done).

## Стек

| Компонент | Выбор | Причина |
|---|---|---|
| Язык | **PHP 8.3** (`/usr/bin/php`, 8.3.32) | Первый чистый-PHP проект портфеля; современный синтаксис, typed properties, union/intersection types |
| Менеджер зависимостей | **Composer 2.2.6** | Автозагрузка PSR-4, dev-зависимости, `composer test` |
| Автозагрузка | **PSR-4**, namespace `Akomyagin\AutowirePHP\` из `src/` | Стандарт экосистемы |
| Тесты | **PHPUnit ^11** | Де-факто стандарт PHP-тестирования |
| Внешние сервисы | **нет** | Чистая библиотека: ни БД, ни Docker, ни сети |

Ядро механики — **Reflection API**: `ReflectionClass`,
`ReflectionMethod::getConstructor()`, `ReflectionParameter`, `ReflectionType`
(`ReflectionNamedType`, `ReflectionUnionType`).

## Архитектура

```
src/
├── Container.php            # публичная точка входа: bind(), get(), has()
├── Exception/              # типизированная иерархия ошибок (появляется по этапам)
│   ├── ContainerException.php          # общий базовый интерфейс/класс
│   ├── NotFoundException.php           # нет биндинга и класс не резолвится
│   ├── NotInstantiableException.php    # abstract/interface без биндинга, private ctor
│   ├── CircularDependencyException.php # цикл (с человекочитаемой цепочкой)
│   └── UnresolvableParameterException.php # параметр без type-hint/дефолта
└── (внутренний Resolver — сначала методы Container, при росте выносится в класс)
tests/
└── ...Test.php             # PHPUnit, зеркалит src/, фикстуры-классы рядом
```

**Инварианты:**
- `Container` — единственный публичный API. Всё остальное — детали реализации.
- AI-агенты/пользователи не конфигурируют контейнер магией: только явный `bind()`
  и автовайринг по type-hint'ам.
- Циклы детектятся **по пути резолвинга** (resolution-stack), не по глобальному
  visited-set (см. Этап 3 и SKILL.md).

## Разбивка по Этапам

### Этап 1 — Базовый Container + ручные биндинги (без autowiring)

**Цель:** контейнер, который умеет отдавать инстанс по явному биндингу и по
конкретному инстанцируемому классу без параметров конструктора.

Задачи:
- `bind(string $abstract, string $concrete)` — сохраняет биндинг
  `interface/abstract -> concrete`.
- `has(string $abstract): bool` — есть ли биндинг.
- `get(string $id)` — разрешает биндинг (если есть), затем инстанцирует класс
  **без параметров конструктора** через `new` / `ReflectionClass::newInstance()`.
- Иерархия исключений: `NotFoundException`, `NotInstantiableException`
  (интерфейс/abstract без биндинга).

**DoD:** биндинг `IFoo -> Foo` резолвится; конкретный класс без ctor-параметров
резолвится напрямую; интерфейс без биндинга кидает осмысленное исключение.
Тесты зелёные, `ruff`-аналога нет — но код проходит `composer test`.

### Этап 2 — Reflection-based автовайринг конструктора

**Цель:** рекурсивно резолвить type-hinted параметры конструктора.

Задачи:
- Через `ReflectionClass::getConstructor()` получить параметры.
- Для каждого `ReflectionParameter` с типом-классом (`ReflectionNamedType`,
  `isBuiltin() === false`) рекурсивно вызвать `get()` для этого типа.
- Собрать инстанс через `newInstanceArgs()`.
- Резолвинг работает и через биндинги: если параметр — интерфейс, сначала
  разворачивается биндинг, потом резолвится реализация (**ленивый резолвинг**).

**DoD:** граф `A(B, C)`, `B(D)` собирается одним `get(A::class)`; параметр-интерфейс
резолвится через биндинг. Пока **без** детекта циклов (цикл на этом этапе даёт
переполнение стека — это ожидаемо и закрывается Этапом 3).

### Этап 3 — Детект циклов через интерфейсы при ленивом резолвинге (ГЛАВНОЕ)

**Цель:** превратить бесконечную рекурсию в осмысленное исключение с
человекочитаемой цепочкой, **включая циклы, которые идут через интерфейсную
косвенность**.

Ключевая идея — resolution-stack:
- Перед резолвингом id (и абстрактного, и конкретного) он **пушится** в стек
  текущего пути резолвинга.
- Если id уже в стеке — это цикл: собрать цепочку из стека, кинуть
  `CircularDependencyException` с читаемым сообщением
  (`IA -> A -> IB -> B -> IA`).
- После успешного резолвинга id **снимается** со стека (path-scoped, а НЕ
  глобальный visited).

Почему это нетривиально: наивный трекинг только *конкретных* классов не поймает
цикл `A -> IB -> B -> IA -> A`, если абстрактные id не участвуют в стеке —
биндинг разворачивается лениво и «прячет» цикл за интерфейсом. Поэтому в стек
кладутся **и абстрактные, и конкретные** идентификаторы.

Задачи:
- Ввести структуру resolution-stack (напр. `array<string,true>` + упорядоченный
  список для сообщения).
- `push`/`pop` вокруг резолвинга как абстрактного id, так и конкретного класса.
- `CircularDependencyException` с методом, отдающим цепочку.
- Гарантировать корректный `pop` при исключении (try/finally), чтобы контейнер
  не «залипал» после ошибки.

**DoD (плотное покрытие):** тесты на:
1. Прямой цикл двух конкретных классов `A(B)`, `B(A)`.
2. Цикл через интерфейсы `A(IB)`, `IB->B`, `B(IA)`, `IA->A`.
3. Самозависимость `A(A)`.
4. Косвенный цикл длиной 3+ через интерфейсы.
5. Сообщение исключения содержит полную читаемую цепочку.
6. После пойманного цикла контейнер остаётся рабочим (стек очищен) — повторный
   `get()` валидного графа проходит.

### Этап 4 — Lifecycle-контроль (singleton vs transient)

**Цель:** управлять временем жизни инстансов.

Задачи:
- Пометка биндинга/класса как `singleton` или `transient` (дефолт выбрать явно и
  задокументировать — предлагается `transient` по умолчанию, `singleton` по
  явной регистрации).
- Кеш singleton-инстансов: повторный `get()` возвращает тот же объект.
- `transient`: каждый `get()` создаёт новый объект.
- Взаимодействие с автовайрингом: singleton-зависимость шарится между
  потребителями в рамках контейнера.

**DoD:** `get()` singleton дважды возвращает тот же инстанс (`assertSame`);
transient — разные (`assertNotSame`); singleton-зависимость общая для двух
разных потребителей.

### Этап 5 — Edge cases конструктора

**Цель:** корректно резолвить нетривиальные сигнатуры конструктора.

Задачи и правила:
- **Nullable-параметр** (`?Foo`) без биндинга и без возможности резолва — если
  разрешимо, резолвим; иначе подставляем `null` (документируем приоритет).
- **Default-значение** builtin-параметра (`int $x = 5`, `string $s = 'a'`) —
  используем default, не пытаясь резолвить builtin.
- **Union types** (`Foo|Bar`) — определить стратегию: пытаться резолвить по
  порядку типов до первого успешного; если ни один не разрешим и нет дефолта —
  `UnresolvableParameterException`.
- **Variadic-параметр** (`Foo ...$foos`) — по умолчанию передаём пустой набор,
  если явного механизма коллекций нет (задокументировать поведение).
- Builtin без дефолта и без биндинга по имени — `UnresolvableParameterException`
  с понятным сообщением (какой класс, какой параметр).

**DoD:** отдельные тесты на каждый case (nullable, default, union, variadic,
неразрешимый builtin), у каждого — детерминированное задокументированное
поведение.

## Тестирование

- PHPUnit, файлы `*Test.php` в `tests/`, зеркалят `src/`.
- Фикстуры (классы `A`, `B`, интерфейсы `IFoo` и т.п.) держим рядом с тестом,
  который их использует, чтобы граф зависимостей был виден в одном месте.
- Каждый Этап добавляет свой набор тестов; Этап 3 — самый плотный.
- Прогон: `composer test` (алиас на `phpunit`) или `vendor/bin/phpunit`.

Подробные конвенции (паттерн работы с Reflection, алгоритм детекта циклов, стиль
тестов) — в `.claude/skills/php-di-dev/SKILL.md`.
