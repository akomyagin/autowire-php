# CLAUDE.md

Guidance for Claude Code when working in the **AutowirePHP** repository.

## О проекте

Framework-agnostic PHP DI-контейнер с reflection-based автовайрингом.
Learning-проект, **первый на чистом PHP** во всём портфеле (до него PHP только
*анализировался* Kotlin-плагином ORM N+1 Radar, но не писался). Не бизнес-идея:
экономика не оценивается, цель — техническая сложность и обучение.

Главная техническая задача — детект циклических зависимостей **через интерфейсы**
при ленивом резолвинге (Этап 3). Подробности — в `docs/` и в
`.claude/skills/php-di-dev/SKILL.md`.

## Раскладка репозитория

| Путь | Содержимое |
|---|---|
| `src/` | Код библиотеки, namespace `Akomyagin\AutowirePHP\`, PSR-4 |
| `tests/` | PHPUnit-тесты, namespace `Akomyagin\AutowirePHP\Tests\` |
| `docs/` | PLAN, TECHNICAL_PLAN, POST_MVP_PLAN |
| `.claude/skills/php-di-dev/` | Конвенции: Reflection-паттерны, алгоритм детекта циклов, стиль тестов |
| `composer.json` / `phpunit.xml` | Зависимости, автозагрузка, конфиг тестов |

Внешних сервисов нет: ни БД, ни Docker, ни сети. Чистая composer-библиотека.

## Команды

```bash
composer install          # установка зависимостей
composer test             # прогон PHPUnit (алиас на phpunit)
vendor/bin/phpunit        # то же напрямую
php -l src/Container.php   # быстрая проверка синтаксиса отдельного файла
```

Перед коммитом: `composer test` должен быть зелёным.

## Конвенции

- **Язык:** документация и subject коммитов — **на русском**; код,
  идентификаторы и комментарии в коде — на английском.
- **Коммиты:** conventional-commit с русским subject, напр.
  `feat(stage3): детект циклов через интерфейсы при ленивом резолвинге`.
  Заканчивать коммит трейлером `Co-Authored-By: Claude`.
- **PHP:** `declare(strict_types=1)` в каждом файле; типизировать всё, что можно;
  `final class` по умолчанию; PSR-4 / PSR-12.
- **Тесты:** PHPUnit, файлы `*Test.php` зеркалят `src/`; фикстурные классы графа
  держать рядом с использующим их тестом. Этап 3 — самый плотный по покрытию.

## Пайплайн разработки (проверенный портфельный)

Для каждого Этапа:

1. **Планирование этапа — Opus 4.8, написание кода — Fable 5.** Opus
   планирует этап по `docs/TECHNICAL_PLAN.md`, Fable 5 реализует код по
   этому плану.
2. **Проверка покрытия и работоспособности — Sonnet 5** (основной чат).
   Проверить тестовое покрытие, дописать/прогнать тесты, убедиться, что всё
   работает (`composer test`).
3. **Независимое ревью — Opus** через Agent-тул (`model: opus`), команда
   `/code-review` на diff ветки.
4. **Цикл исправлений — до 3 итераций** между ревью и правками.
5. **Commit + push + PR** (conventional-commit, русский subject) в `master`.

## Git-workflow

- Новая ветка от `master` на **каждый Этап**: `stage-<N>/<topic>`.
- Работа в ветке → PR → merge в `master`.
- Не коммитить и не пушить без явного запроса пользователя.

## Ключевые инварианты

- `Container` — единственный публичный API; всё остальное — детали реализации.
- Детект циклов — **по пути резолвинга** (resolution-stack с абстрактными И
  конкретными id), а НЕ по глобальному visited-set. Иначе циклы через интерфейсы
  не ловятся. Это ядро проекта — см. SKILL.md.
- Runtime-reflection only в MVP: attributes-конфигурация и компиляция графа —
  строго после MVP (`docs/POST_MVP_PLAN.md`).
